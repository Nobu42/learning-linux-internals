# 03_Process_Scheduler

## やったこと

プロセススケジューラの基本を、CPU負荷プログラムを使って観察した。

主に見たもの:

- `real`, `user`, `sys`
- `taskset` によるCPU固定
- 1CPU上で複数プロセスを動かしたときの挙動
- 複数CPUで並列実行したときの挙動
- タイムスライスの可視化
- `nice` 値による優先度の違い
- `%user`, `%nice`
- 平均ターンアラウンドタイム
- スループット

## CPU負荷プログラム

```python
#!/usr/bin/python3

NLOOP=100000000

for _ in range(NLOOP):
    pass
```

これはI/Oやsleepをせず、CPUを使い続けるプログラム。

## real, user, sys

```bash
time ./load.py
```

例:

```text
real    0m3.308s
user    0m3.304s
sys     0m0.004s
```

意味:

- `real`: 開始から終了までの実時間
- `user`: ユーザー空間でCPUを使った時間
- `sys`: カーネル空間でCPUを使った時間

CPU負荷処理では、1プロセスなら `real` と `user` がほぼ同じになる。

## 1CPUに固定して複数プロセスを動かす

`multiload.sh` では、`-m` なしの場合にCPU 0へ固定する。

```bash
taskset -p -c 0 $$ >/dev/null
```

意味:

- `taskset`: プロセスが使えるCPUを指定する
- `-p`: 既存プロセスを対象にする
- `-c 0`: CPU 0だけを使う
- `$$`: 現在のシェルスクリプト自身のPID

子プロセスも基本的にCPU affinityを引き継ぐため、複数の `load.py` がCPU 0を取り合う。

## 1CPU固定時の結果

```bash
./multiload.sh 1
./multiload.sh 2
./multiload.sh 3
```

結果の傾向:

```text
1プロセス: real 約3.3秒
2プロセス: real 約6.3秒
3プロセス: real 約9.4秒
```

各プロセスの `user` はだいたい3秒前後。

大事なこと:

```text
各プロセスが必要とするCPU時間はほぼ同じ。
しかし1つのCPUを取り合うため、待ち時間が増えて real が伸びる。
```

## 複数CPUで動かす

`-m` を付けるとCPU 0固定をしない。

```bash
./multiload.sh -m 2
```

例:

```text
real    約3.2s
user    約3.2s
```

2つのプロセスが別CPUで並列に動けるため、`real` が2倍にならない。

外側からさらに `time` すると:

```bash
time ./multiload.sh -m 2
```

例:

```text
real    0m3.351s
user    0m6.542s
sys     0m0.012s
```

大事なこと:

```text
複数CPUで並列実行すると、user の合計が real を超えることがある。
```

## CPU数の確認

`lscpu` で確認したCPU構成:

```text
CPU: 4
コアあたりのスレッド数: 2
ソケットあたりのコア数: 2
ソケット数: 1
```

つまり:

```text
1ソケット × 2物理コア × 2スレッド = 4論理CPU
```

Linuxからは CPU 0, 1, 2, 3 として見える。

## タイムスライスの可視化

```bash
for i in 1 2 3 ; do ./sched.py $i ; done
```

生成されるファイル:

```text
sched-1.jpg
sched-2.jpg
sched-3.jpg
```

`sched.py` の流れ:

```text
CPU 0に固定する
子プロセスを複数 fork する
各子プロセスがCPU負荷を実行する
進捗時刻を 0.data, 1.data, 2.data に保存する
plot_sched.py でグラフ化する
```

`.data` ファイルの形式:

```text
経過時間[ミリ秒]    進捗[%]
```

データ上で時間が飛んでいる箇所は、そのプロセスがCPUをもらえず、別プロセスが動いていた時間と考えられる。

## コンテキストスイッチ

1つのCPU上で複数の実行可能プロセスがある場合、スケジューラは実行するプロセスを切り替える。

```text
プロセスAが少し進む
プロセスBが少し進む
プロセスCが少し進む
またプロセスAが進む
```

この切り替えがコンテキストスイッチ。

タイムスライスの可視化グラフでは、各プロセスの進捗が交互に進む様子として見える。

## nice値

`nice` 値を変えて、CPUのもらいやすさを観察した。

```bash
./sched-nice.py 5
```

意味:

```text
負荷処理0: nice値 0
負荷処理1: nice値 5
```

結果例:

```text
0.data: 最後が約142ms
1.data: 最後が約262ms
```

nice値が大きいプロセスのほうが遅く終わった。

大事なこと:

```text
nice値が大きいほど優先度は低くなり、CPU時間をもらいにくくなる。
```

## sarでniceを見る

```bash
nice -n 5 taskset -c 0 ./inf-loop.py &
sar -P 0 1 1
```

結果:

```text
%nice 100.00
```

意味:

```text
CPU 0が、nice値付きのユーザー空間プロセスに100%使われていた。
```

`%user` と `%nice` はどちらもユーザー空間のCPU時間だが、`%nice` はnice値が変更されたプロセスのCPU時間を表す。

## sar -P

```bash
sar -P 0 1 1
```

意味:

- `-P`: processor単位で見る
- `0`: CPU 0を見る
- `1 1`: 1秒間隔で1回取得する

`taskset -c 0` でCPU 0に固定した実験では、`sar -P 0` が確認に向いている。

## 性能測定

`cpuperf.sh` で、プロセス数を1から最大値まで増やしながら性能を測定した。

```bash

```

生成されるファイル:

```text

```

`cpuperf.data` の形式:

```text

```

実行結果:

```text

3	9.783	0.303
4	12.686	0.311
5	16.331	0.305
6	18.880	0.314
```

## ターンアラウンドタイムとスループット

ターンアラウンドタイム:

```text
プロセス開始から終了までにかかった時間
```

平均ターンアラウンドタイム:

```text
複数プロセスのターンアラウンドタイムの平均
```

スループット:

```text
単位時間あたりに完了できたプロセス数
```

今回の結果では、プロセス数が増えるほど平均ターンアラウンドタイムは伸びた。

一方、スループットはおおむね:

```text
約0.30 プロセス/秒
```

で大きくは増えなかった。

## 性能測定からわかったこと

1CPU固定では、CPU負荷プロセスを増やしてもCPUそのものの処理能力は増えない。

```text
プロセス数を増やす
  ↓
各プロセスの待ち時間が増える
  ↓
平均ターンアラウンドタイムが伸びる
  ↓
スループットはほぼ一定
```

大事なこと:

```text
1つのCPUでCPU負荷処理をたくさん同時に動かしても、
全体の処理能力はほとんど増えない。
ただし、各プロセスの完了までの時間は長くなる。
```
## 複数CPUでの性能測定

`-m` を付けて、CPU 0固定をせずに性能測定した。

```bash
./cpuperf.sh -m 8
```

結果:

```text

2	3.690	0.533
3	4.685	0.539
4	6.960	0.557
5	8.561	0.576
6	10.329	0.557
7	12.137	0.528
8	14.858	0.524
```

形式:

```text
<プロセス数> <平均ターンアラウンドタイム[秒]> <スループット[プロセス/秒]>
```

## 1CPU固定との比較

1CPU固定では、プロセス数を増やしてもスループットはおおむね一定だった。

```text
約0.30 プロセス/秒
```

一方、`-m` ありでは複数CPUに分散して実行できるため、スループットが上がった。

```text
1プロセス: 0.292
2プロセス: 0.533
3プロセス: 0.539
4プロセス: 0.557
5プロセス: 0.576
```

つまり、複数CPUを使うことで、単位時間あたりに完了できるプロセス数が増えた。

## 読み取り

この環境は論理CPUが4つある。

```text
CPU 0, 1, 2, 3
```

そのため、`-m` ありでは複数の `load.py` が複数CPUに分散して実行できる。

ただし、プロセス数を増やせば無限に速くなるわけではない。

理由:

- 論理CPU数には上限がある
- 他のプロセスもCPUを使っている
- スケジューリングのオーバーヘッドがある
- 同一物理コア上のSMTでは、完全に2倍の性能にはならない

今回の結果では、スループットはおおむね:

```text
約0.5〜0.58 プロセス/秒
```

あたりで頭打ちになっている。

## 大事なこと

```text
1CPU固定:
  プロセス数を増やしてもスループットはほぼ増えない

複数CPU:
  複数プロセスを並列実行できるためスループットが上がる
  ただしCPU数やオーバーヘッドの制約で頭打ちになる
```

## 03_Process_Scheduler のまとめ

今回の一番大事な理解:

```text
スケジューラは、実行可能なプロセスにCPU時間を分配する。
```

実験で見えたこと:

- CPUバウンドな処理では `real` と `user` が近くなる
- 1CPUに複数プロセスを置くと `real` が伸びる
- 複数CPUを使えると並列に進み、`user` の合計が `real` を超えることがある
- タイムスライスにより、複数プロセスが少しずつ進む
- コンテキストスイッチで実行プロセスが切り替わる
- nice値を上げるとCPUをもらいにくくなる
- プロセス数を増やすと平均ターンアラウンドタイムは伸びる
- 1CPU固定ではスループットはほぼ一定になる
