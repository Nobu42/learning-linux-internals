## fork() の高速化: Copy-on-Write

`fork()` は親プロセスを複製して子プロセスを作る。

ただし、Linuxでは `fork()` 時に親の物理メモリをすべて即コピーするわけではない。

```text
fork直後:
  親子で同じ物理ページを共有する

どちらかが書き込む:
  そのページだけコピーする
```

これを **Copy-on-Write** という。

## cow.py

`cow.py` では、100MiBの匿名メモリを確保し、親プロセスで全ページにアクセスした後に `fork()` した。

その後、子プロセスでも同じメモリに書き込み、メモリ使用量やページフォルトを観察した。

結果:

```text
*** 子プロセス生成前 ***
親プロセスのメモリ関連情報
  RSS  MAJFL  MINFL
112408     0  27081

*** 子プロセス生成直後 ***
子プロセスのメモリ関連情報
  RSS  MAJFL  MINFL
110236     0    632

*** 子プロセスによるメモリアクセス後 ***
子プロセスのメモリ関連情報
  RSS  MAJFL  MINFL
110308     0  26670
```

## Copy-on-Writeで見えたこと

100MiBのページ数:

```text
100MiB / 4KiB = 25600ページ
```

親プロセスでは、100MiBにアクセス済みなので `MINFL` が約27000になった。

`fork()` 直後、子プロセスも約110MiBのRSSを持っているように見える。

しかし、システム全体のメモリ使用量は大きく増えていない。

これは:

```text
親子が同じ物理ページを共有している
```

ため。

子プロセスが書き込みを行うと、Copy-on-Writeが発生する。

```text
子が共有ページに書き込む
  ↓
カーネルがそのページをコピーする
  ↓
子はコピー後の自分専用ページへ書き込む
```

その結果:

- システム全体のメモリ使用量が増える
- 子プロセスの `MINFL` が増える
- `MAJFL` は0のまま

ディスクI/Oを伴わないため、メジャーフォールトではなくマイナーフォールトになる。

## なぜforkが高速になるか

もし `fork()` のたびに親のメモリを全部コピーしていたら、大きなプロセスの複製は重い。

Copy-on-Writeでは:

```text
fork時:
  ページテーブルを複製する
  物理ページは共有する

書き込み時:
  必要なページだけコピーする
```

そのため、`fork()` 自体は高速になる。

特に以下のような流れと相性が良い。

```text
fork()
  ↓
exec()
```

子プロセスがすぐ別プログラムに置き換わる場合、親のメモリを丸ごとコピーする必要がない。

## プロセス間通信

プロセスは基本的に別々のアドレス空間を持つ。

そのため、通常の変数は親子プロセス間でも共有されない。

## 通常の変数は共有されない

`non-shared-memory.py`:

```python
#!/usr/bin/python3

import os
import sys

data = 1000

print("子プロセス生成前のデータの値: {}".format(data))

pid = os.fork()
if pid < 0:
    print("fork()失敗しました", file=sys.stderr)
elif pid == 0:
    data *= 2
    sys.exit(0)

os.wait()
print("子プロセス終了後のデータの値: {}".format(data))
```

実行結果:

```text
子プロセス生成前のデータの値: 1000
子プロセス終了後のデータの値: 1000
```

子プロセス側で `data *= 2` をしても、親プロセス側の `data` は変わらない。

理由:

```text
fork後、親と子は別々のプロセス空間を持つ。
通常の変数は共有されない。
```

## 共有メモリ

`shared-memory.py` では `mmap.MAP_SHARED` を使って、親子プロセス間で共有できるメモリを作った。

```python
shared_memory = mmap.mmap(-1, PAGE_SIZE, flags=mmap.MAP_SHARED)
```

共有メモリに整数を書き込むため、整数を8バイトのバイト列に変換して保存した。

```python
shared_memory[0:8] = data.to_bytes(8, byteorder)
```

子プロセスは共有メモリから値を読み、2倍にして書き戻した。

```python
data = int.from_bytes(shared_memory[0:8], byteorder)
data *= 2
shared_memory[0:8] = data.to_bytes(8, byteorder)
```

実行結果:

```text
子プロセス生成前のデータの値: 1000
子プロセス終了後のデータの値: 2000
```

大事なこと:

```text
通常の変数:
  fork後は親子で別々
  子が変更しても親には反映されない

共有メモリ:
  親子で同じメモリ領域を見る
  子が変更すると親からも見える
```

## シグナル

`dd` に `SIGUSR1` を送って、進捗表示をさせた。

```bash
dd if=/dev/zero of=test bs=1 count=10M &
DDPID=$!
kill -SIGUSR1 $DDPID
```

`$!` は直前にバックグラウンド実行したプロセスのPID。

`dd` は `SIGUSR1` を受け取ると、途中経過を表示する。

例:

```text
3612942+0 レコード入力
3612942+0 レコード出力
3612942 bytes copied ...
```

大事なこと:

```text
シグナルは、プロセスに非同期イベントを通知する仕組み。
```

`SIGUSR1` はユーザー定義用シグナルで、プログラム側が独自用途に使える。

## パイプ

```bash
free | awk '(NR==2){print $2}'
```

これは `free` の標準出力を `awk` の標準入力につないでいる。

```text
free
  ↓ パイプ
awk
```

`awk '(NR==2){print $2}'` の意味:

- `NR==2`: 2行目だけ処理する
- `print $2`: 2列目を表示する

`free` の2行目は `Mem:` の行なので、2列目は total メモリ。

大事なこと:

```text
パイプは、あるプロセスの標準出力を、別プロセスの標準入力につなぐプロセス間通信。
```

## ソケット

ソケットもプロセス間通信の一種。

種類:

```text
Unixドメインソケット:
  同じマシン内のプロセス間通信に使う

  ネットワーク越しに信頼性のある通信をする

  ネットワーク越しに軽量なデータグラム通信をする
```

## 排他制御

`inc.sh`:

```bash
#!/bin/bash

TMP=$(cat count)
echo $((TMP + 1)) > count
```

これは:

```text
countを読む
1足す
countに書く
```

という処理。

1回だけなら正しく増える。

しかし、1000個同時に実行すると:

```bash
echo 0 > count
for ((i=0;i<1000;i++)) ; do ./inc.sh & done
for ((i=0;i<1000;i++)) ; do wait; done
cat count
```

結果:

```text
4
```

期待値は1000だが、実際には大きく小さい値になった。

理由:

```text
複数プロセスが同時に count を読む
同じ値に1を足して書き戻す
更新が失われる
```

これを **競合状態**、race condition という。

## クリティカルセクション

同時に複数プロセスが実行すると壊れる処理区間を **クリティカルセクション** という。

今回のクリティカルセクション:

```bash
TMP=$(cat count)
echo $((TMP + 1)) > count
```

この部分は、複数プロセスから同時に実行されると結果が壊れる。

## アトミック処理

アトミック処理とは:

```text
途中で割り込まれない不可分な処理
```

理想的には:

```text
countを読む
1足す
書き戻す
```

が1つのアトミックな処理なら安全。

しかし `inc.sh` の処理は複数の操作に分かれており、アトミックではない。

## 間違ったロック

`inc-wrong-lock.sh`:

```bash
#!/bin/bash

while : ; do
    if [ ! -e lock ] ; then
        break
    fi
done

touch lock

TMP=$(cat count)
echo $((TMP + 1)) > count

rm -f lock
```

意図:

```text
lock ファイルがなければ進む
lock ファイルを作る
count を更新する
lock ファイルを消す
```

しかしこれは安全ではない。

問題は:

```bash
if [ ! -e lock ]
```

と:

```bash
touch lock
```

が別々の処理であること。

複数プロセスが同時に:

```text
lock がない
```

と判断してしまう可能性がある。

その結果、1000回実行しても:

```text
count = 38
```

になった。

大事なこと:

```text
ロックの存在確認と作成がアトミックでないと、ロックとして壊れる。
```

## flockによる正しい排他制御

`inc-lock.sh`:

```bash
#!/bin/bash

flock lock ./inc.sh
```

`flock` はカーネルのファイルロック機構を使う。

意味:

```text
lock ファイルに排他ロックを取る
ロックを取れたプロセスだけ ./inc.sh を実行する
./inc.sh が終わったらロックを解放する
```

1000回同時に実行した結果:

```text
count = 1000
```

期待通りになった。

大事なこと:

```text
共有データを更新する処理では、正しい排他制御が必要。
```

## ロックなし・間違ったロック・flockの比較

```text
ロックなし:
  count = 4

間違ったロック:
  count = 38

flock:
  count = 1000
```

「ロックっぽいもの」を自作しても、ロック取得自体がアトミックでないと壊れる。

## カーネルスレッド

```bash
ps -eLF
```

意味:

- `-e`: 全プロセスを表示
- `-L`: スレッドも表示
- `-F`: 詳細形式で表示

主な列:

- `UID`: 実行ユーザー
- `PID`: プロセスID
- `PPID`: 親プロセスID
- `LWP`: Light Weight Process。スレッドID
- `C`: CPU使用率の目安
- `NLWP`: そのプロセス内のスレッド数
- `SZ`: 仮想メモリサイズの目安
- `RSS`: 実メモリ使用量
- `PSR`: 最後に実行されたCPU番号
- `STIME`: 開始時刻
- `TTY`: 端末
- `TIME`: 累積CPU時間
- `CMD`: コマンド

## カーネルスレッドの見分け方

例:

```text
root  2   0  2  ... [kthreadd]
root  3   2  3  ... [rcu_gp]
root 13   2 13  ... [ksoftirqd/0]
root 15   2 15  ... [migration/0]
root 42   2 42  ... [writeback]
root 104  2 104 ... [kswapd0]
```

`CMD` が `[ ... ]` で囲まれているものは、主にカーネルスレッド。

特徴:

- `PPID` が2のものが多い
- `SZ` が0
- `RSS` が0
- `TTY` が `?`
- `CMD` が `[名前]`

## kthreadd

```text
[kthreadd]
```

`kthreadd` はカーネルスレッドを管理する親のような存在。

多くのカーネルスレッドは:

```text
PPID = 2
```

になっている。

## なぜSZ/RSSが0なのか

カーネルスレッドは、普通のユーザープロセスのような独立したユーザー空間を持たない。

そのため:

```text
SZ  = 0
RSS = 0
```

のように見える。

これは「メモリを全く使っていない」という意味ではなく:

```text
ユーザープロセスとしての仮想メモリ空間やRSSを持たない
```

と考える。

## 代表的なカーネルスレッド

```text
[kworker/...]
```

カーネルの汎用作業スレッド。
遅延実行されるカーネル内処理などを担当する。

```text
[ksoftirqd/0]
```

CPU 0 用のソフトIRQ処理スレッド。
ネットワークやI/Oの後処理などに関係する。

```text
[migration/0]
```

CPU 0 に関係するタスク移動用スレッド。

```text
[rcu_sched]
[rcu_gp]
```

RCUというカーネル内同期機構に関係する。

```text
[kswapd0]
```

メモリ回収、スワップ、ページ回収に関係する。

```text
[writeback]
```

dirty pageをディスクへ書き戻す処理に関係する。

```text
[jbd2/sda2-8]
```

ext4のジャーナリングに関係する。

## ユーザープロセスのスレッド

例:

```text
root  658  1  658 ... NLWP 11 ... /usr/lib/snapd/snapd
root  658  1  826 ... NLWP 11 ... /usr/lib/snapd/snapd
root  658  1  835 ... NLWP 11 ... /usr/lib/snapd/snapd
```

同じ `PID=658` の `snapd` に複数の `LWP` がある。

意味:

```text
PID:
  プロセスID

LWP:
  スレッドID

NLWP:
  そのプロセスに属するスレッド数
```

ユーザープロセスのスレッドは、同じPIDを共有しつつ、LWPが異なる。

## 05_Process_Management_Advanced のまとめ

今回の大事な理解:

```text
プロセスは基本的に独立したメモリ空間を持つ。
必要に応じて、共有メモリ・パイプ・ソケット・シグナルなどでプロセス間通信を行う。
```

重要ポイント:

- `fork()` はCopy-on-Writeにより高速化されている
- fork直後は親子で物理ページを共有する
- 書き込み時にページ単位でコピーされる
- 通常の変数は親子プロセス間で共有されない
- `MAP_SHARED` の共有メモリを使うと親子で値を共有できる
- シグナルはプロセスへ非同期イベントを通知する
- パイプは標準出力と標準入力をつなぐプロセス間通信
- ソケットはローカルまたはネットワーク越しのプロセス間通信
- 共有データの更新には排他制御が必要
- クリティカルセクションは同時実行されると壊れる処理区間
- アトミックでないロック実装は壊れる
- `flock` を使うと安全に排他制御できる
- カーネルスレッドはカーネル内部の仕事をする実行単位

全体の流れ:

```text
fork()
  ↓
Copy-on-Writeで親子が物理ページを共有
  ↓
通常の変数は親子で別々
  ↓
共有したい場合はIPCを使う
  ↓
共有データを更新する場合は排他制御が必要
```
